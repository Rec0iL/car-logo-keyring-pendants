hobbyist 3D-printing / identification use (keyring pendant engraving). Not for redistribution
or resale. Every logo below is tagged on Commons as "public domain / ineligible for copyright"
(simple geometric shapes/text under the threshold of originality) but ALSO carries a trademark
notice — the brand names/emblems remain registered trademarks of their respective companies in
most jurisdictions. This paper trail records exactly where each file came from and what Commons
says about its status; it is not a legal opinion.

## Notes on suitability for engraving

- Most files above are clean single-color/flat emblems or bold wordmarks — good candidates for
  extrusion onto a 3mm disc.
- Peugeot and Jeep ended up as wordmark-only because Commons did not have a clean SVG of their
  current pictorial emblem (Peugeot's post-2021 lion head, Jeep's 7-slot grille) — only raster
  versions exist for those. If a pictorial mark is preferred for those two, they'd need to be
  manually vectorized/traced from a raster source (out of scope for this pass).
- Chevrolet's SVG is very compact (371 bytes) — verified as a complete, well-formed single-path
  bowtie outline, not a broken/truncated file.
- Subaru's SVG is unusually large (~1.5 MB) for a flat logo; likely contains embedded metadata
  or unnecessarily complex paths. It renders correctly but may be worth simplifying/optimizing
  (e.g. with svgo) before use in the disc generator.

## New brands added 2026-07-22: Citroën, SEAT, Cupra

All three were checked for the Ford-style "background shape swallows detail" problem (list distinct
fill colors from both `fill="..."` attributes and `style="fill:...;"`, style taking precedence)
before finalizing.

- **Citroën** (`logos/citroen.svg`, id `citroen`): the chosen file ("Citroen 2021 (alternate).svg")
  is a *single* compound `<path>` (no explicit fill attribute → default black, `fill-rule` not set
  so effectively nonzero, no gradients). One path, one color — structurally there is nothing to
  layer/swallow, so no `keep_fill`/`drop_fill` override is needed or added. Other Citroën files on
  Commons were also inspected: "Citroën 2021.svg" and "Citroen 2022.svg" are the full lockup
  (wordmark + chevrons together, also flat black, no gradients — fine but not "mark alone" as
  requested); "Citroën-Logo.svg" is a small favicon-style badge with a red square background and a
  white chevron drawn on top (`fill="#ed1c24"` background + `fill="#fff"` foreground) — this WOULD
  have the Ford problem, but it wasn't used, so no filter was needed. "Citroën.svg" is a wordmark
  only. Went with the mark-alone flat file as it best matches the request.
- **SEAT** (`logos/seat.svg`, id `seat`): "SEAT Logo from 2017.svg" — the current stylised "S"
  emblem, 2 `<path>` elements, both `fill="#19191b"` (same single dark color), no gradients, no
  layering issue. No override added. ("SEAT S.A. logo.svg" was also checked but is the corporate
  holding-company wordmark, not the car-brand "S" mark, so it wasn't used.)
- **Cupra** (`logos/cupra.svg`, id `cupra`): the task flagged this as likely gradient-heavy since
  Cupra's official mark uses a copper/bronze gradient. Checked two candidates: "Cupra.svg" (emblem
  + "CUPRA" wordmark together, single black default fill, no gradients) and "Cupra symbol.svg"
  (emblem alone, single `<path>`, no fill attribute → default black, no gradients). Went with
  "Cupra symbol.svg" since the task asked for the shield/diamond emblem specifically. Both files
  turned out to be genuinely flat single-path uploads — no gradient defs at all, so no
  `keep_fill`/`drop_fill` override needed. (If a copper-gradient version is ever substituted in
  later, it would need re-analysis — a single-path gradient shape cannot be fixed by
  `keep_fill`/`drop_fill` since there's no other flat-colored shape to isolate.)

## Replacements 2026-07-22: BMW, Mercedes-Benz, Škoda

Re-evaluated per instructions because the original 3 files were glossy gradient-shaded "3D badge"
renders unsuitable for flat print geometry.

- **BMW** (`logos/bmw.svg`): replaced "File:BMW.svg" (~20 gradient defs, chrome-shaded roundel)
  with **"File:BMW logo (gray).svg"**. Read the raw SVG: 4 `<path>` elements total, zero
  `<linearGradient>`/`<radialGradient>` defs — `path21` (`fill:#0166b1` blue, two opposite
  quadrants), `path21-3` (`fill:#ffffff` white, other two quadrants), `path14` (`fill:#6f6f6f` grey,
  the outer ring), and one more grey path drawing the "BMW" wordmark letters. The blue/white
  quadrants tile the disc edge-to-edge (not nested), so unioning them all reconstructs the full
  roundel + wordmark silhouette correctly — no `keep_fill`/`drop_fill` override needed. Depicts the
  same 1997-2020 roundel era as the file it replaces.
- **Mercedes-Benz** (`logos/mercedes_benz.svg`): replaced "File:Mercedes-Benz Star 2022.svg"
  (~15 gradient defs, scraped from mercedes-benz.com footer CSS) with **"File:Mercedes-Benz Star
  (1969-1986, 2025-).svg"**. Read the raw SVG: a single flat black `<path>` (multiple subpaths via
  one `d` string, `style="fill:#000"`), no gradients at all, 1000×1000 viewBox — a compact
  compound-path rendering of the classic three-pointed-star-in-a-ring badge. Bonus: per its own
  Commons title this is the variant in use 1969-1986 and again from 2025 onward, i.e. it is the
  actual current mark as of today (2026).
- **Škoda** (`logos/skoda.svg`): **no flat replacement found.** Checked
  "File:Skoda-Auto-Logo-2011-present.svg" (3840×2160, 654 KB) — this turned out to be an
  auto-traced raster render (548 `<path>` elements, hundreds of near-duplicate greyscale shading
  fills like `#3A3838`, `#292827`, `#5C5959`...) simulating gradients via discrete flat-color
  layers rather than actual `<linearGradient>` defs — worse than the current file, not usable.
  Searched Commons categories for Škoda/Škoda Auto logos and found nothing else resembling a clean
  flat winged-arrow emblem.

  Instead, applied a targeted `keep_fill` override to the *existing* file. Read the raw SVG
  structure in full: it has 37 `<linearGradient>` defs, but only 4 `<path>`s actually reference one
  via `fill="url(#...)"` — these are the outer bezel/rim shading circle and two small
  highlight/dimple shapes. Separately there are flat-colored paths: the "ŠKODA" wordmark
  (`fill:#141414`), a black ring circle (`fill:#141414`, radius ~81 in local units), a white disc
  behind it (`fill:#ffffff`, radius ~74.6, nested inside the black ring), a white "eye" dot
  (`fill:#ffffff`, radius ~14.7), and the green winged-arrow shape itself (`fill:#42bd3b`, one
  `<path>`, the actual detail we want).

  **Problem:** the black ring circle and the wordmark share the exact same hex color
  (`#141414`), so there is no way to `keep_fill`/`drop_fill` a combination that keeps the wordmark
  AND drops the ring — they're indistinguishable by fill color. And dropping only the `"gradient"`
  sentinel isn't safe: the white disc (radius 74.6) is fully nested inside the black ring
  (radius 81) which is fully nested inside... i.e. if the black ring is kept, union of ring + white
  disc + green wing simply collapses back to the outer black circle silhouette (identical to the
  original Ford problem) and the wing disappears — this fails the "recognizable, complete
  silhouette" bar.

  So the safe option is `"keep_fill": ["#42bd3b"]`, which keeps *only* the green winged-arrow path
  and drops everything else (gradients, ring, white disc, eye dot, and — as an accepted
  trade-off — the wordmark, since it shares a color with the ring and can't be cleanly separated).
  The wing itself ("Škoda Winged Arrow" is literally the name Škoda uses for this mark on its own
  site) is a complete, standalone, recognizable emblem on its own — consistent with how this
  library already treats e.g. Chevrolet (bowtie only, no wordmark) — so this is judged safe.
  **Flagging for manual check anyway**: nobody in this pipeline can render OpenSCAD, so it is not
  verified whether the small "eye" sub-path inside the green path's `d` string (a ~19-unit-radius
  circular loop) renders as a solid dot or a punched-through hole once isolated — either is a minor
  cosmetic detail, not a missing chunk, but worth a human glance in the OpenSCAD GUI. If the
  wordmark is wanted back on the pendant alongside the wing, that requires either a manually
  recolored/re-exported SVG (so ring and text no longer collide on hex value) or accepting the
  wing-only result and adding a separate text engraving in OpenSCAD directly.

## Added 2026-07-22: Citroën (2022) as a second variant

Added `logos/citroen_2022.svg` (id `citroen_2022`) alongside the existing `citroen.svg`, at the
user's request, without touching or removing the original entry.

Downloaded from `https://commons.wikimedia.org/wiki/File:Citroen_2022.svg` (Special:FilePath
redirect, 200 OK, well-formed XML, no gradients — a single flat-black compound path for the mark
plus separate letter-shape paths spelling "CITROEN"). PD (below threshold of originality) +
trademark notice, same framing as every other file in this document.

**Important caveat, flagged rather than silently resolved:** the filename says "2022," but reading
its actual path geometry makes that questionable. This file's mark is a *circular* badge — one
compound path whose first subpath is a circle outline, with wave/chevron-like cutout shapes as
further subpaths inside it — plus a full "CITROEN" wordmark. That does not match the real-world
2022 Citroën rebrand, which (to the best of the analysis available here, with no way to render
either file and compare visually) is a flat, frameless, two-piece open chevron mark with no circular
border and no wordmark attached. The **existing** `citroen.svg` (sourced last round from "Citroen
2021 (alternate).svg" — two bare chevron shapes, no frame) actually looks like the closer match to
that real current design, despite its file's name suggesting 2021.

In short: Commons' file-naming years for Citroën are not reliable evidence of which design is
actually current. Both files are kept as separate manifest entries (`citroen` and `citroen_2022`)
precisely so a human can render both in OpenSCAD and pick/keep whichever actually matches the
current logo — this document does not assert `citroen_2022` supersedes `citroen`.

## Replaced 2026-07-22: Subaru

The original "Subaru logo (transparent).svg" (row 17 above, ~1.5MB) was replaced with a
user-supplied file (`logos/subaru-12.svg`, renamed to `logos/subaru.svg`) — smaller (3.6KB) and
structurally better suited to multi-color (base + inlay) printing: a clean "SUBARU" wordmark, the
oval ring, and the six-star cluster, each as ordinary flat-black paths.

**No confirmed Commons source URL for this file** — it was added directly to the project rather
than sourced via the search process used for everything else in this document. If you have the
original source (Commons file title, or wherever it came from), it's worth adding here for the
same provenance/license paper trail as the other 24 files; until then this entry is missing that
column deliberately rather than guessing one.

**Ford-style background-swallow problem, found and fixed the same way as Ford/Škoda:** the file's
first `<path>` is `fill="#fff" d="M0 0h192.756v192.756H0V0z"` — a rectangle covering the *entire*
192.756×192.756 canvas. Every other shape (wordmark, ring, stars) defaults to black (no `fill`
attribute) and sits inside that rectangle's area, so importing everything as-is collapses to a
plain square, the same failure mode as Ford's navy oval. Fixed with `"drop_fill": ["#ffffff"]` in
the manifest entry, which removes just that background rect and keeps the black wordmark/ring/star
paths — verified via `svg_content_bbox()`: unfiltered bbox is the full 192.76×192.76 canvas;
filtered bbox is 182.5×67.1, matching the actual wordmark+star-cluster content.
